Тема моей дипломной работы "Разработка системы выявления индивидуальных параметров утомления по сигналу ЭКГ с применением машинного обучения". Я собираю данные с себя. Набор данных за один день представляет из себя несколько записей ЭКГ за час до и через час после тренировки, различные признаки самочувствия (субъективные шкалы, давление, скорость реакции, время сна), оценки и метрики тренировочной нагрузки. Запись производится с помощью простого датчика на AD8232 в одном отведении (левая и правая подключичная область и референс внизу живота справа). Качество записей отличается ото дня ко дню. Различные уровни выскочастотных шумов, иногда попадаются артефакты - скачки baseline или даже весьма продолжительный клиппинг (полный пропуск R-пика).

&#x20;План записи на день

&#x20;before (до тренировки):

&#x20;sit (60 сек сидя в покое)

&#x20;stand (60 сек стоя)

&#x20;squat (60 сек сразу после 40 приседаний)

&#x20;breath\_in (задержка дыхания на вдохе)

&#x20;breath\_out (задержка дыхания на выдохе)

&#x20;long (дополнительно 5 мин сидя в покое, под спокойные видео)

&#x20;И то же самое в after (после тренировки) - пока after не должен использоваться в моей конфигурации протокола

&#x20;Всё сохраняется соответствующими файлами - примеры в \\row\_ECG\_feature\_extractor\\input\\sample\_data



Центральная идея:

Если ЭКГ-признак, измеренный до тренировки, улучшает прогноз самочувствия на стандартизированной тренировке относительно базовой модели, значит этот признак можно рассматривать как индивидуально информативный параметр утомления/готовности.



Методические ограничения:

* Система не ищет универсальные маркеры утомления. Она выявляет индивидуальные параметры, информативные для конкретного пользователя, конкретного типа тренировок и конкретного протокола сбора данных.
* Ограничена методической чистотой выбора таргета
* Может выдавать почти случайный результат на маленькой выборке



Предварительный технический пайплайн разрабатываемой системы:

1\. Пользователь задаёт протокол сбора данных.

2\. Система извлекает признаки из записей ЭКГ.

3\. Система строит составные признаки физиологических реакций.

4\. Система объединяет ЭКГ, контекст и target в дневной датасет.

5\. Система исключает дни, не соответствующие условиям стандартизированной тренировки.

6\. Система анализирует пригодность target.

7\. Система предварительно фильтрует признаки по качеству и минимальной связи с target.

8\. Система обучает baseline-модель по контекстным признакам.

9\. Система автоматически добавляет ЭКГ-признаки и оценивает прирост качества.

10\. Система устраняет избыточность признаков.

11\. Система формирует итоговый набор индивидуальных параметров.

12\. Система строит финальную модель прогноза самочувствия.



Существующая структура:

* protocol.yaml - файл протокола, основное средство конфигурирования работы системы. Основная идея дать пользователю гибкость в подстройке работы системы исходя из своего видения процедуры сбора данных, построения признаков, выбора таргета модели, а также оптимальных методических решений.
1. .1\_row\_ECG\_feature\_extractor/ - модуль извлечения сырых признаков из записей ЭКГ
2. .2\_ECG\_feature\_derived/ - модуль для расчёта составных признаков, задаваемых в файле протокола
3. 2\_context\_features/ - рассчитывает составные признаки из контекстных ежедневных данных, может идти параллельно с ECG\_feature\_derived
4. 3\_dataset\_builder - собирает данные в один файл
5. 4\_feature\_analyses (устарел) - содержит в себе эксперименты до смены концепции работы системы, требуется обновление. Должен представлять из себя предварительный фильтр для модели.

Далее должен идти ML-блок, пока не реализован. В директории каждого модуля лежат папки inputs и outputs - входные и выходные данные для этого модуля.
